3 에이전트 체계
모든 세션은 목적에 따라 A·B·C 중 하나로 발동된다. 각 에이전트는 자신의 영역 밖 파일을 임의로 수정하지 않는다.
Agent A
실행 에이전트
loop-b-execution-agent.txt
  • Protocol 파일 작성·수정 (유일한 권한)
  • Node App 구현 — API Route, buildSystemPrompt()
  • AGENT B 결함 보고서 기반 수정 (HIGH → MID → LOW)
  • handoff 파일 작성 후 AGENT B에 인계
  • 자기 자신은 PASS/FAIL을 발급하지 않음
Agent B
검증 에이전트
loop-b-verification-agent.txt
  • 항상 Fresh 세션으로 시작 (기억 없음)
  • V1 Loop A → V2 품질점수 → V3 구현 → V3.5 코드리뷰 → V4 시뮬레이션
  • PASS/FAIL 판정 발급 (유일한 권한)
  • 결함 보고서: Priority × Layer 분류
  • 블로킹 결함이 있으면 PASS 불가

Section 03
Loop A — Protocol 정적 검증 파이프라인
Protocol 파일을 실제 API 호출 없이 정적으로 검증한다. AGENT A가 신규 Protocol 작성 직후, AGENT B가 V1 단계에서 실행한다.
Loop A — 4개 검증 체크
1
CHECK 1 — 구조 완전성
6개 필수 섹션 존재 여부 + 하위 필드 전수 확인
  • SYSTEM (노드명, 버전)
  • GOAL (1-3문장)
  • CONTEXT (Ontological Status, Operational Logic, Immutable Constants)
  • ROLE
  • ACTION PROTOCOL (Pre-Step + Step 1..N)
  • COMPLIANCE CHECK (Pre-flight, Post-generation, Failure Mode ≥2개 IF-THEN)
2
CHECK 2 — 간결성
토큰 수 ≤ 50,000 / 중복 탐지
  • 전체 Protocol + Knowledge Docs 합계 모델 컨텍스트 25% 이내
  • 반복 지시 / 불필요한 예시 중복 탐지
3
CHECK 3 — 내부 일관성
ACTION PROTOCOL ↔ COMPLIANCE CHECK 정렬
  • 각 Step에 대응하는 COMPLIANCE CHECK 존재 여부
  • Failure Mode 완전성
  • Knowledge Doc 지시 충돌 여부
4
CHECK 4 — 오염 저항성 (Stage A2)
5개 오염 패턴 방어 확인 — 4/5 이상 통과 필수
  • Input Pass-Through 방어
  • Geometry Modification 방어
  • Step Skipping 방어
  • Abstract Command 비번역 방어
  • Hallucination 방어
✓ PASS — 4개 모두 통과 → Loop B 진입
✗ FAIL — 즉시 보고 → AGENT A 수정 요청

품질 기준 (QUALITY_SCORE.md)
Protocol Compliance Score(PCS) ≥ 90 이상 달성 시에만 배포를 승인한다.
Protocol Compliance
0기준: 90+100
Pre-Step 존재·비어있지 않음
각 Step에 검증 가능한 출력 기술
COMPLIANCE CHECK가 모든 Step에 매핑
PCS = (CC 매핑 Step / 전체 Step) × 100
Immutable Constants
CONTEXT에 선언된 모든 불변 상수
COMPLIANCE CHECK Post-generation에 등재
Geometry Sanctuary 원칙 준수
건물 비율·구조·형태는 어떤 노드에서도 변경 불가
Boundary Resolution
범위 밖 입력 처리 Failure Mode ≥ 1
비가시·모호 영역 처리 Failure Mode ≥ 1
"원본 반환 금지" 명시
거부 또는 원본 반환 대신 반드시 추론하여 완성
Output-Specific
product-spec 출력 계약 필드 반영
